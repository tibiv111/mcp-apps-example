R Shiny — integration options

Embedding R Shiny inside an MCP App resource is not a single decision — the right path depends on the host's iframe sandbox, whether Shiny needs to be reachable from the user's browser, and how much engineering you're willing to spend. The cards below show four approaches running against the same backing app ({{ shiny_url }}), so you can compare in place.

A · DIRECT NESTED IFRAME

Iframe a hosted Shiny URL directly

Simplest possible wiring: an <iframe src="{{ shiny_url }}"> lives inside the MCP shell. Works in the browser preview at /ui/shell. Blocked by Claude's host CSP (frame-src 'self' blob: data:) because the iframe loads a different origin than the shell.

B · OPEN IN NEW TAB

Punch out of the iframe sandbox entirely

A button that does window.open(SHINY_URL, '_blank'). The new tab loads at the top level — no MCP sandbox, no CSP path through the shell. Always works. Tradeoff: you lose the in-canvas feel and the user context-switches to another tab.

target: {{ shiny_url }}
C · SAME-ORIGIN REVERSE PROXY

Front Shiny with this service so the iframe is same-origin

FastAPI proxies /shiny-proxy/* (HTTP + WebSocket) to the Shiny service. The iframe src is /shiny-proxy/ — same-origin to the shell, so the host's frame-src 'self' permits it. Works inside Claude at the cost of real engineering: Shiny's HTML uses absolute paths that have to be rewritten, and the WebSocket upgrade has to be pumped bidirectionally. Source: app/shiny_proxy.py.

D · URL-FORM MCP RESOURCE

Let the host open its own iframe (documentation only)

Instead of returning inline HTML from resources/read, the MCP server returns a URL pointer. The host opens its own iframe at the Shiny URL — no nesting, no shell-side CSP path. This is the cleanest option spec-wise, but client support is narrow: today Claude renders inline HTML resources reliably but treats URL-form resources more conservatively. 

// Hypothetical resources/read response — not yet implemented in this repo.
{
  "contents": [{
    "uri": "ui://nav-ai/shiny",
    "mimeType": "text/uri-list",
    "text": "{{ shiny_url }}",
    "_meta": {
      "io.modelcontextprotocol/ui": {
        "displayMode": "inline",
        "externalUrl": "{{ shiny_url }}"
      }
    }
  }]
}

Implementing it would add a new tool (e.g. launch_shiny) that returns this shape from app/mcp/router.py's resources/read handler. Whether it actually renders depends on the host honoring the URL form.
