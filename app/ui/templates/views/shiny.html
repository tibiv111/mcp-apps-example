R Shiny — integration options

Embedding R Shiny inside an MCP App is not a single decision. Four approaches are shown below, each pointing at the same Shiny service ({{ shiny_url }}). Status badges report what we've empirically observed in browser preview (loading /ui/shell directly) versus inside Claude's MCP host, which enforces frame-src 'self' blob: data: and a sandbox without allow-popups.

Diagnosis (so far) In Claude, the shell HTML renders under an opaque host origin, so 'self' in frame-src 'self' is not our service URL — every `<iframe src=>` from inside the shell is blocked, even back to our own domain. Card D (URL-form resource) the host rejects outright with Unsupported UI resource content format. Card E's srcdoc trick slips past frame-src and Shiny now renders, but the iframe's connect-src 'self' https://nav-ai-mock-mcp.onrender.com blocks the WebSocket to wss://…/shiny-proxy/websocket/ (CSP3's https→wss scheme flex isn't honoured here), so reactive outputs stay empty. The fix landed in the resource's connectDomains: the wss form is listed explicitly now. Card F takes the cleanest architectural path: expose Shiny as a separate MCP server, let Claude mount it as a peer iframe alongside this workspace.

C · SAME-ORIGIN REVERSE PROXY

Front Shiny with this service so the iframe is same-origin

✓ BROWSER PREVIEW ✗ BLOCKED IN CLAUDE

FastAPI proxies /shiny-proxy/* (HTTP + WebSocket) to the Shiny service so the iframe target is on our domain — see app/shiny_proxy.py. In a normal browser this passes frame-src 'self'. In Claude it still fails: the shell document is loaded under the host's opaque origin, so even our own service URL counts as cross-origin from inside the shell.

E · SERVER-SIDE EMBED (INLINE HTML)

Serve Shiny as an inline-HTML MCP resource

✓ BROWSER PREVIEW ~ RENDERS IN CLAUDE (NO LIVE DATA UNTIL WSS FIX REDEPLOYS)

What we observed: Card E's Shiny shell rendered inside Claude (path rewriter + srcdoc trick worked) but every KPI/plot/table stayed empty. Console showed: Connecting to 'wss://nav-ai-mock-mcp.onrender.com/shiny-proxy/websocket/' violates the following Content Security Policy directive: "connect-src 'self' https://nav-ai-mock-mcp.onrender.com". The iframe's CSP, derived from the resource's connectDomains, only allowed the https:// form — browsers don't auto-extend to wss:// in practice. Fix: the resource now lists both schemes explicitly. Once redeployed, the live data should flow.

The MCP server fetches Shiny's HTML, rewrites every absolute path to point at {{ base_url }}/shiny-proxy/, injects a <base href> and a WebSocket-constructor shim, then returns it with mimeType: text/html;profile=mcp-app — the same MIME the NAV AI shell uses. Claude renders this the same way it renders the workspace. The trick is the WS shim: Shiny computes its WebSocket URL from location.host, which is empty in an inline-HTML iframe, so we monkey-patch window.WebSocket to force every connection at our proxy. Source: app/shiny_proxy.py::fetch_embedded_html.

D · URL-FORM MCP RESOURCE

Let the host open its own iframe

✓ SPEC-CORRECT ✗ HOST REJECTS

The MCP server returns a URL-form resource ({{ '"' }}mimeType{{ '"' }}: {{ '"' }}text/uri-list;profile=mcp-app{{ '"' }}) with _meta.ui.externalUrl set to {{ shiny_url }}. A compliant host would open its own iframe at the URL — the shell's CSP would never be in the picture. Empirically, Claude rejects this MIME today. Calling the tool from chat surfaces the error: Unsupported UI resource content format: {"uri":"ui://nav-ai/shiny","mimeType":"text/uri-list;profile=mcp-app",…} So Card D is wire-format-correct and unambiguously not the path for today's host. F · SEPARATE MCP SERVER

Expose Shiny as its own MCP server, peer to NAV AI

✓ ARCHITECTURALLY CLEAN ? NEEDS CLAUDE CONFIG CHANGE

Instead of trying to mount Shiny inside this NAV AI shell, expose it as a dedicated MCP server endpoint at /shiny-mcp. The user adds this URL as a second MCP server in Claude — each MCP server gets its own iframe slot, so the Shiny dashboard mounts as a peer of the NAV AI workspace rather than fighting for the same surface. The path rewriter + WS shim still do their work inside that iframe's resource, but the srcdoc trick from Card E isn't needed: Claude mounts the second server's resource as a fresh iframe naturally.

MCP server URL {{ base_url }}/shiny-mcp
Tool launch_shiny_dashboard
Resource ui://shiny/dashboard
MIME text/html;profile=mcp-app

To try it:

  1. Add {{ base_url }}/shiny-mcp as a connector in Claude (separate from the existing NAV AI server).
  2. In chat, ask Claude: "open the Shiny dashboard". Claude should invoke launch_shiny_dashboard on the new server.
  3. The Shiny dashboard mounts in its own iframe — independent of this NAV AI workspace. Both stay live; both are addressable from the same chat.

Source: app/shiny_mcp.py (full MCP server in ~150 LOC, reuses fetch_embedded_html from shiny_proxy).
