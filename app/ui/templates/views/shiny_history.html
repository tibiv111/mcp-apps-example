R Shiny — integration attempts

Five approaches we tried before landing on the working pattern. Each points at the same Shiny service ({{ shiny_url }}). Kept here so the failure modes and the reasoning behind the final design stay visible — the working version is one click back.

Empirical findings A/B/C: host enforces frame-src 'self' blob: data: and a parent sandbox without allow-popups; the shell's origin isn't our service, so even same-origin tricks fail. D: host rejects URL-form resources outright — Unsupported UI resource content format. E: srcdoc slips past frame-src and Shiny renders inside the existing shell. Live data required listing the wss:// form explicitly in connectDomains (the https:// form alone doesn't permit WebSocket here).

Bottom line: Card E renders end-to-end in Claude today. D's MIME is rejected outright. A–C are unrecoverable inside the current host iframe sandbox. The working version (Card F + the MCP App handshake) lives on the main Shiny page.

B · OPEN IN NEW TAB

Punch out of the iframe sandbox entirely

✓ BROWSER PREVIEW ✗ BLOCKED IN CLAUDE

window.open(SHINY_URL, '_blank'). In Claude, fails with Blocked opening … in a new window because the request was made in a sandboxed frame whose 'allow-popups' permission is not set — the parent (host-owned) sandbox is missing allow-popups; nothing the resource HTML sets can override it.

target: {{ shiny_url }}
C · SAME-ORIGIN REVERSE PROXY

Front Shiny with this service so the iframe is same-origin

✓ BROWSER PREVIEW ✗ BLOCKED IN CLAUDE

FastAPI proxies /shiny-proxy/* (HTTP + WebSocket) to the Shiny service so the iframe target is on our domain — see app/shiny_proxy.py. In a normal browser this passes frame-src 'self'. In Claude it still fails: the shell document is loaded under the host's opaque origin, so even our own service URL counts as cross-origin from inside the shell.

D · URL-FORM MCP RESOURCE

Let the host open its own iframe

✓ SPEC-CORRECT ✗ HOST REJECTS

The MCP server returns a URL-form resource ({{ '"' }}mimeType{{ '"' }}: {{ '"' }}text/uri-list;profile=mcp-app{{ '"' }}) with _meta.ui.externalUrl set to {{ shiny_url }}. A compliant host would open its own iframe at the URL — the shell's CSP would never be in the picture. Empirically, Claude rejects this MIME today. Calling the tool from chat surfaces the error: Unsupported UI resource content format: {"uri":"ui://nav-ai/shiny","mimeType":"text/uri-list;profile=mcp-app",…}

E · SERVER-SIDE EMBED (INLINE HTML)

Serve Shiny as an inline-HTML MCP resource

✓ BROWSER PREVIEW ✓ RENDERS IN CLAUDE
